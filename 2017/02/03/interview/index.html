加班汪爬坡记

2017-02-03 • 标签: java

近段时间准备跳槽，因此自己总结一下遇到的面试问题，顺带也总结一下自己的2016年~

2016小结

关键字: 充实 学习 苦涩

16年初在一家外企, 这是我的第一家工作单位, 工作非常轻松, 朝九晚五周末双休从不加班, 以至于我看到新闻上程序员加班严重, 甚至出现过劳猝死的情况时, 开始怀疑我们干的是同一个世界的程序员么~

当时感觉再待下去已无法学到更多的技能了, 出于对未来职业发展的考虑, 选择了离职, 原因说通俗点就是工作太轻松了=_=, 但刚毕业半年的我技术真的很菜, 当时工作不是很好找, 多轮面试后进入了一家小型创业互联网公司, 有多小型呢, 这么说吧, 进公司的时候我是公司仅有的三个Java程序员之一, 其中有一个只比我早来两天… 因为公司之前后端的主力语言是Python, 我进来的时候刚准备开始将后端转到Java服务, 毕竟Java有着完善生态体系, 各种各样取之不尽、拿来即用的开源或封闭的轮子, 作为主力语言构建后端服务是再适合不过了。

进新公司后接到的第一个大的任务、也是公司后端重构的第一个任务—负责公司的基础账号系统包括ERP管理在内的设计开发, 基本上公司后续的开发任务, 以及其他服务系统都是将这个账号系统作为基础服务, 因此要求比较急, 期望我两三周的样子给出一个基本的后台系统, 作为一个小菜鸡当时我说实话有点蒙, 而且恰好当时招我进公司的唯一的一个资深的Java准备离职了.. 连问的都没得问了, 只能自己蒙着头乱撞了, 当时开发初期真的非常累, 天天十一二点都是常事, 周末不时来加个班。

当然这段时间也是我16年成长最快的一段时间, 我学会git操作命令让我代码版本变得可控; 接触到了mongodb, redis 这样的nosql数据库; 对Spring SpringMVC Mybatis 各种配置不再是照搬, 开始熟悉原理; 独立负责一台linux主机环境搭建维护, 学会了linux各种基础命令以及shell的编写; 以及16年后半年对这个系统的维护升级, 也提升了自己对sql规范的重视; 业余时间在同事的安利下开始写起了博客, 开始把自己平常工作生活的学到总结下来写到博客中来。

可惜的是这样充实的学习工作状态并没有持续多久, 十月份国庆回来后开始被一些无脑的需求压得喘不过气, 来自上头的需求一开始就不对等, 根本无法讨论需求的合理性, 对接的实习产品只不过是个传声筒, 再加上无理的加班要求, 不对等的薪资回报, 因此在做完最后一个小系统后, 17年初我又回到了自由市场, 只不过这一次相比上次匆忙的离职, 信心大了不是一点半点, 所以面了一周, 拿了3个offer, 基本就确定下家了, 剩下的时间和女朋友去婺源浪了一圈, 美滋滋~

最后, 17年希望继续充实, 继续码自己想要的代码, 生命不息, 学习的脚步永远不止。

面试总结

答案仅为个人观点，并非标准

  1. mybatis运行原理，以及$与#区别，mybatis缓存机制
    mybatis是一个orm框架，实际上是对jdbc进行了封装，完成了从查询对象关系映射、结果集的解析、动态sql的参数替换、懒加载以及缓存机制，并使用连接池管理数据库连接，mybatis根据xml配置生成sqlSessionFactory，再根据注解或xml里的sql生成sqlSession去执行映射的sql做增删改操作以及事务提交，操作完成会关闭会话。

    $使用的jdbc的Statement操作sql，存在sql注入等风险; #为PreparedStatement进行sql预编译，具有更好的安全性与性能。

    一级缓存作用域session无需配置自动支持，二级缓存作用域mapper，二级缓存面对关联查询仍有问题，缓存没有实时更新被关联表数据

  2. Spring autowired与Resource注解
    这两个都是Spring Ioc注入对象的注解，autowired默认byType注入，resource属于java的注解，Spring支持注入对象，默认为byName注入

  3. 1000个线程对同一个Integer对象加一，怎么同步数据，需要volatile关键字申明吗

    不需要volatile关键字，volatile只保证每个线程读到的是主内存中最新的值，并不具有原子性。

  4. 表A字段姓名，性别，省份，查询每个省份下男女人数

  5. redis 数据结构，对象序列化
    redis支持String、哈希、list、set、有序集合;
    可以将对象序列化成字节码或者json(可以选择不同序列化方案)写入内存

  6. 事务的四个特性
    ACID
    原子性（Atomicity）：事务是一个原子操作，由一系列动作组成。事务的原子性确保动作要么全部完成，要么完全不起作用。
    一致性（Consistency）：一旦事务完成（不管成功还是失败），系统必须确保它所建模的业务处于一致的状态，而不会是部分完成部分失败。
    隔离性（Isolation）：可能有许多事务会同时处理相同的数据，因此每个事务都应该与其他事务隔离开来，防止数据损坏。
    持久性（Durability）：一旦事务完成，无论发生什么系统错误，它的结果都不应受到影响

  7. spring事务
    配置事务管理bean(jdbc, jpa, Hibernate不同平台提供的事务管理机制)，spring并不管理事务，而是提供事务管理器委托不同平台来实现

    一、事务的传播行为propagation

    使用spring的transactional注解默认的传播行为为Propagation.REQUIRED(必须运行在一个事务内)，具体详解: http://www.mamicode.com/info-detail-1248286.html

    二、事务的隔离级别isolation level

    • 脏读（Dirty reads）—— 脏读发生在一个事务读取了另一个事务改写但尚未提交的数据时。如果改写在稍后被回滚了，那么第一个事务获取的数据就是无效的。
    • 不可重复读（Nonrepeatable read）—— 不可重复读发生在一个事务执行相同的查询两次或两次以上，但是每次都得到不同的数据时。这通常是因为另一个并发事务在两次查询期间进行了更新。

    • 幻读（Phantom read）—— 幻读与不可重复读类似。它发生在一个事务（T1）读取了几行数据，接着另一个并发事务（T2）插入了一些数据时。在随后的查询中，第一个事务（T1）就会发现多了一些原本不存在的记录。

      隔离级别:

    • ISOLATION_DEFAULT 使用后端数据库默认的隔离级别
    • ISOLATION_READ_UNCOMMITTED 最低的隔离级别，允许读取尚未提交的数据变更，可能会导致脏读、幻读或不可重复读
    • ISOLATION_READ_COMMITTED 允许读取并发事务已经提交的数据，可以阻止脏读，但是幻读或不可重复读仍有可能发生
    • ISOLATION_REPEATABLE_READ 对同一字段的多次读取结果都是一致的，除非数据是被本身事务自己所修改，可以阻止脏读和不可重复读，但幻读仍有可能发生
    • ISOLATION_SERIALIZABLE 最高的隔离级别，完全服从ACID的隔离级别，确保阻止脏读、不可重复读以及幻读，也是最慢的事务隔离级别，因为它通常是通过完全锁定事务相关的数据库表来实现的

  8. mvn install、package、deploy的区别
    package仅仅是打包到项目target下，而install还会打包到本地仓库供其他项目使用，deploy则是打包到远程仓库

  9. Java并发包下有些什么类
    ConcurrentHashMap (支持并发)，ThreadPoolExecutor (线程池)，BlockingQueue (阻塞队列ArrayBlockingQueue/LinkedBlockingQueue)，
    CyclicBarrier (栅栏)、AtomicInteger、AtomicLong、AtomicBoolean(原子操作)
